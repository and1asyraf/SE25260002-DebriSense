Example of Test Case Template: 
Test Case ID: UT001
Test Case Name: Login User 
Description: This test case verifies whether the system allows users to log in with valid credentials, ensures authentication and directs to the dashboard page.
ID: UT001-TC-01
Test Input: User inputs valid email and password. User clicks on “Login” button.
Expected Result: User is logged into the application and directed to the dashboard page.
Pass Criteria: User is logged into the application and directed to the dashboard page
Fail Criteria: User is not logged into the application.
ID: UT001-TC-02
Test Input: User inputs unregistered email. User clicks on “Login” button.
Expected Result: Error message of unregistered email is displayed. User is unable to log into the application
Pass Criteria: Error message of unregistered email is displayed. User is unable to log into the application.
Fail Criteria: Error message of unregistered email is not displayed. User is logged into the application
ID: UT001-TC-03
Test Input: User inputs invalid password. User clicks on “Login” button.
Expected Result: Error message of wrong email and password combination is displayed. User is unable to log in.
Pass Criteria: Error message of wrong email and password combination is displayed. User is unable to log in.
Fail Criteria: Error message of wrong email and password combination is not displayed. User is logged into the application

Test Case ID: IT001
Test Case Name: Backend to Database Integration
Description: Verifies that data submitted through frontend API endpoints accurately persists in the relational database safely.
ID: IT001-TC-01
Test Scenario: Submit new Hotspot Report via API with all valid fields.
Expected Result: Database HotspotReport table inserts a new row matching JSON inputs.
Pass Criteria: ORM commit succeeds, queried data matches perfectly.
Fail Criteria: Database throws integrity or constraint error.
ID: IT001-TC-02
Test Input: Submit Hotspot Report with an invalid River ID (foreign key violation).
Expected Result: Database rejects the insert, backend catches error and returns HTTP 400/500 gracefully.
Pass Criteria: Error handled gracefully, system does not crash.
Fail Criteria: System crashes, exposing raw SQL traceback to client.

Test Case ID: IT002
Test Case Name: Weather API Integration
Description: Verifies that the system can successfully fetch external weather data and handle failures.
ID: IT002-TC-01
Test Input: Trigger automated DRI update job for valid river coordinates.
Expected Result: External API returns HTTP 200, JSON is parsed correctly into the DRI calculation function.
Pass Criteria: Payload parsed without key errors, DRI is updated.
Fail Criteria: API timeout crashes the job execution.

Pass Criteria: Loop continues for other rivers, system remains stable.
Fail Criteria: Entire background scheduler halts.
